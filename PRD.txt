1. Executive Summary
================================================================

BloodHound AI is an open-source offensive security research tool that augments BloodHound Community Edition with a large language model (LLM) reasoning layer. It automatically analyzes Active Directory relationship graphs, surfaces non-obvious attack paths that standard shortest-path queries miss, and generates executable command sequences mapped to MITRE ATT&CK — all within an isolated lab environment.

The tool is designed for red team operators, penetration testers, and security researchers building portfolio projects for roles in offensive security. It demonstrates that LLMs can reason meaningfully over graph-structured AD data, closing the gap between raw BloodHound output and actionable attack narratives.

Critically, BloodHound AI operates exclusively on pre-collected BloodHound JSON data. It does not touch live systems, does not perform enumeration, and does not exfiltrate data. Its sole input is a BloodHound export; its sole output is a structured attack path report for authorized use.

================================================================
2. Problem Statement
================================================================

2.1 The BloodHound Gap

BloodHound is the industry standard for Active Directory attack path visualization. Every mature red team uses it. However, the tool surfaces paths through pre-defined Cypher queries — shortest paths, DCSync paths, Kerberoastable users — queries a human wrote in advance.

The gap: real AD environments are messy. Novel attack paths emerge from unusual combinations of ACL misconfigurations, trust relationships, group memberships, and delegation settings that no pre-written query anticipates. A skilled human analyst can find these paths by reasoning over the graph, but this takes hours and requires deep expertise.

2.2 Current Workflow Pain

Pain Point           | Current State                                | Impact
---------------------|----------------------------------------------|----------------------------------------
Path Discovery       | Manual GUI clicking through BloodHound graph | Hours per engagement, expert-dependent
Novel Path Detection | No systematic approach beyond default queries| Non-obvious paths missed entirely
Command Generation   | Analyst writes commands from memory          | Slow, inconsistent, error-prone
ATT&CK Mapping       | Manual lookup after the fact                 | Often skipped due to time pressure
Report Generation    | Manual write-up from notes                   | Bottleneck at end of every engagement

5. Technical Architecture
================================================================

5.1 System Overview

BloodHound AI is a Python CLI application with four primary components: a Neo4j data ingestion layer, a graph-to-text serializer, an LLM reasoning engine, and a structured report generator.

5.2 Component Breakdown

Component 1: Neo4j Ingestor
- Connects to local BloodHound Neo4j instance via bolt://localhost:7687
- Executes Cypher queries to extract nodes (users, computers, groups, GPOs, OUs, domains) and relationships (MemberOf, AdminTo, HasSession, CanRDP, DCSync, GenericAll, WriteDACL, etc.)
- Normalizes output into a structured Python dataclass graph representation
- Alternatively accepts BloodHound JSON export files directly — no live Neo4j required

Component 2: Graph Serializer
- Converts the graph dataclass into chunked natural language descriptions optimized for LLM context windows
- Implements intelligent chunking — large AD environments exceed context limits, so the serializer prioritizes high-value nodes (Domain Admins, DCs, Tier 0 assets) and their relationships
- Produces structured JSON schema describing node attributes, edge types, and relationship metadata
- Handles deduplication and relationship normalization

Component 3: LLM Reasoning Engine
- Sends serialized graph chunks to Claude or GPT-4 API with a structured system prompt
- System prompt instructs the model to: identify multi-hop attack paths, reason about privilege escalation chains, flag unusual ACL combinations, and prioritize by impact/stealth/complexity
- Implements a multi-pass approach: initial path discovery pass, then a refinement pass that validates logical consistency of each path
- Returns structured JSON output: list of attack paths, each with steps, required tools, detection likelihood, and ATT&CK mappings

Component 4: Report Generator
- Consumes structured JSON from the reasoning engine
- Generates a Markdown and optional PDF report with: executive summary, prioritized path list, per-path command sequences, ATT&CK navigator layer JSON, and detection recommendations
- ATT&CK navigator layer is automatically generated and can be imported directly into the ATT&CK Navigator web tool

5.3 Tech Stack

Component          | Technology                              | Rationale
-------------------|-----------------------------------------|--------------------------------------------------
Language           | Python 3.11+                            | Universal in security tooling, rich ecosystem
AD Data Source     | BloodHound CE + Neo4j / JSON export     | Industry standard, already in every red team workflow
Graph Queries      | neo4j Python driver + Cypher            | Native Neo4j interface, full query flexibility
LLM API            | Anthropic Claude API (claude-sonnet-4-6)| Strong reasoning over structured data, large context
CLI Framework      | Click or Typer                          | Clean argument parsing, help text generation
Report Output      | Jinja2 templates + markdown             | Flexible, version controllable, CI/CD friendly
ATT&CK Integration | mitreattack-python library              | Official MITRE library for technique validation
Testing            | pytest + intentionally misconfigured AD | Reproducible, documented test environment

6. Core Features
================================================================

6.1 Feature: Novel Path Discovery

The primary differentiator. Beyond default BloodHound shortest-path queries, the LLM reasons over the full graph to find:
- Multi-hop paths through unexpected intermediary objects (e.g., GenericAll on a GPO that applies to a group that AdminTo a DC)
- Shadow admin chains — users with Write permissions on high-value group objects who are not explicit members
- Cross-domain trust abuse paths that traverse forest trust relationships
- ACL inheritance chains where permissions cascade through OU structure in non-obvious ways
- Kerberos delegation abuse combinations (unconstrained + resource-based + protocol transition chains)

6.2 Feature: Executable Command Generation

For each identified path, the tool generates exact commands ready to copy-paste in an authorized lab:
- PowerView commands for enumeration steps
- Impacket command strings for remote exploitation (secretsdump, GetUserSPNs, etc.)
- Rubeus commands for Kerberos attacks (AS-REP roasting, ticket requests, S4U abuse)
- BloodHound CE custom query strings to visualize the specific path in the GUI
- All commands annotated with expected output and failure conditions

6.3 Feature: Prioritization Scoring

Each path receives a composite score across three dimensions:

Dimension  | Description                                                                 | Weight
-----------|-----------------------------------------------------------------------------|-------
Impact     | Highest privilege achievable at path end (DA, EA, local admin, data access) | 40%
Stealth    | Estimated detection likelihood based on technique visibility in common SIEMs | 35%
Complexity | Number of steps, required tools, prerequisite access, likelihood of success  | 25%

6.4 Feature: MITRE ATT&CK Integration
- Every technique in every path mapped to ATT&CK tactic, technique, and sub-technique
- ATT&CK Navigator layer JSON generated automatically — importable to navigator.attack.mitre.org
- Detection query suggestions per technique (Sigma rules where applicable)
- Links to relevant ATT&CK technique pages for research reference

6.5 Feature: Detection Guidance

For every offensive technique surfaced, the report includes a corresponding defensive section:
- Windows Event IDs to monitor for this technique
- Sigma rule stub the blue team can implement
- Remediation recommendation (ACL fix, configuration hardening, etc.)
- This dual-use design is intentional — it makes the ethical framework airtight and increases real-world value
